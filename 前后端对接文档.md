# 前后端对接文档

## 1. 文档目的

本文档用于说明智慧交通系统中 **后端如何向前端传输可视化所需数据**，以及 **前端在接收后需要如何渲染**。当前约定采用：

- **后端负责**：视频分析、模型推理、业务逻辑判断、坐标和结果数据推送；
- **前端负责**：实时视频播放、接收后端消息、在视频上叠加绘制框、渲染热力图与告警面板。

即：

> 后端不把检测框直接画进视频流里，而是将检测结果和框坐标通过 Socket.IO 推送给前端；前端收到后，在视频播放器上叠加绘制。

---

## 2. 当前通信方式

### 2.1 实际通信协议

虽然设计文档中将结果推送描述为 WebSocket，但当前项目实际实现为：

- **后端**：Flask-SocketIO
- **前端**：socket.io-client

当前实时消息主事件名：

```text
analysis_result
```

前端应监听：

```js
socket.on('analysis_result', (data) => {
  // 处理后端推送的分析结果
})
```

### 2.2 当前主运行入口

当前真正用于前端联调的后端入口为：

- `cloud/stream_receiver/main_server.py`
- `cloud/stream_receiver/video_processor.py`

这条链路负责：

- 车辆检测（vehicle_detection）
- 拥堵热力图（traffic_density）
- 违停跟踪与告警（illegal_parking）
- 道路异常检测（road_anomaly）

### 2.3 车牌识别当前运行链路

当前车牌识别逻辑在：

- `cloud/stream_receiver/api_server_sandbox.py`
- `cloud/stream_receiver/plate_recognition_processor.py`

这条链路目前是单独的 HTTP 服务，用于测试 **plate YOLO + LPRNet**。如果以后要让前端的统一主画面同时接收车牌框与车牌识别结果，建议后续把车牌识别并入主运行链路或在前端额外接入其结果源。

---

## 3. 前端接收的数据分为两类

### 3.1 业务事件流（用于面板、列表、事件流）

后端当前已经通过 `analysis_result` 推送以下业务事件：

- `vehicle_detection`
- `plate_recognition`
- `traffic_density`
- `illegal_parking`
- `road_anomaly`
- `system_status`

这些事件用于：

- 右侧事件流
- 车辆检测面板
- 车牌识别面板
- 违停告警列表
- 道路异常列表
- 系统状态显示
- 拥堵热力图

### 3.2 视频叠加快照（用于在视频上画框）

为了让前端能够做到：

- 当前帧检测到车辆就画车辆框
- 当前帧存在禁停区配置就画禁停区多边形
- 当前帧检测到违停车就画违停车框
- 当前帧检测到道路异常就画异常物品框
- 当前帧检测到车牌就画车牌框（主链路字段已预留，当前通常为空）
- 当前帧没有检测到对象就不画框

后端已新增统一的：

```text
video_overlay
```

事件类型。此事件同样通过：

```text
analysis_result
```

推送给前端。前端在收到 `event_type === 'video_overlay'` 时，应将其视为 **当前处理帧的完整画框快照**。

## 4. 统一消息基础字段

所有实时结果消息建议遵循统一基础结构：

```json
{
  "event_type": "...",
  "timestamp": "2026-07-09T16:30:00.123Z",
  "device_id": "mobile_001",
  "status": "normal",
  "data": {},
  "bbox": []
}
```

字段说明：

- `event_type`：事件类型
- `timestamp`：事件产生时间
- `device_id`：视频源设备 ID
- `status`：状态，例如 `normal / detected / warning`
- `data`：业务字段主体
- `bbox`：单目标事件中的框坐标（若适用）

---

## 5. 视频叠加快照 `video_overlay`

### 5.1 推送目的

前端如果只接收零散的单条 `vehicle_detection` / `illegal_parking` / `road_anomaly` 事件，会遇到两个问题：

1. 不容易判断什么时候清空旧框；
2. 不同类别框不一定属于同一时刻的同一帧。

因此后端已补充 **按处理帧聚合的画框快照消息**，供前端专门用于视频叠加绘制。

### 5.2 当前推荐结构

```json
{
  "event_type": "video_overlay",
  "timestamp": "2026-07-09T16:30:00.123Z",
  "device_id": "mobile_001",
  "status": "normal",
  "stream_size": {
    "width": 720,
    "height": 1280
  },
  "data": {
    "no_parking_zones": [
      {
        "zone_id": "no_parking_A",
        "name": "禁停区A",
        "polygon": [[53, 310], [580, 310], [608, 529], [50, 532]],
        "threshold": 2,
        "label": "禁停区A"
      }
    ],
    "vehicles": [
      {
        "track_id": 3,
        "bbox": [120, 320, 210, 440],
        "label": "vehicle 0.79",
        "confidence": 0.79
      }
    ],
    "plates": [
      {
        "bbox": [150, 400, 190, 420],
        "label": "京B6789T",
        "confidence": 0.83,
        "track_id": 3
      }
    ],
    "illegal_parking": [
      {
        "bbox": [120, 320, 210, 440],
        "track_id": 3,
        "stay_time": 3.8,
        "threshold": 2,
        "status": "warning",
        "label": "track 3 3.8s"
      }
    ],
    "road_anomalies": [
      {
        "bbox": [360, 220, 420, 280],
        "label": "unknown_object",
        "status": "warning",
        "affected_lane": "lane_1",
        "duration_frames": 20
      }
    ]
  }
}
```

### 5.3 字段含义

#### 顶层字段
- `event_type = video_overlay`：表明这是视频叠加快照
- `timestamp`：该处理帧对应时间
- `device_id`：视频源
- `stream_size.width / height`：后端处理时的视频原始尺寸，前端做坐标缩放时要用到

#### `data.vehicles`
- 用于画**车辆框**
- 当前主链路已能提供

#### `data.no_parking_zones`
- 用于画**禁停区多边形**
- 当前主链路已能从 `illegal_parking_config.json` 的 `no_parking_zones` 配置中提供
- 前端以半透明橙色区域叠加显示

#### `data.plates`
- 用于画**车牌框**
- 当前字段结构已预留，便于未来将车牌识别并入主运行链路后直接使用；主链路中通常为空

#### `data.illegal_parking`
- 用于画**违停框**
- 当前主链路已能提供

#### `data.road_anomalies`
- 用于画**道路异常框**
- 当前主链路已能提供

### 5.4 前端渲染规则

前端收到 `video_overlay` 后，应该：

1. 清空当前画布上的旧框；
2. 读取 `no_parking_zones / vehicles / plates / illegal_parking / road_anomalies`；
3. 逐类绘制：禁停区画多边形，车辆/违停/异常/车牌画矩形框；
4. 如果某一类是空数组，则这一帧不画该类图层。

> 必须按帧刷新，不能只增量累加画框，否则会残留上一帧的旧框。

## 6. 业务事件详细结构

### 6.1 车辆检测事件 `vehicle_detection`

#### 后端当前会传

```json
{
  "event_type": "vehicle_detection",
  "timestamp": "2026-07-09T16:30:00.123Z",
  "device_id": "mobile_001",
  "bbox": [120, 320, 210, 440],
  "status": "detected",
  "data": {
    "vehicle_id": 1,
    "track_id": 3,
    "vehicle_type": "vehicle",
    "confidence": 0.79
  }
}
```

#### 前端用途
- 车辆检测记录列表
- 事件流
- 车辆框叠加（也可由 `video_overlay.vehicles` 提供）

---

### 6.2 车牌识别事件 `plate_recognition`

#### 后端当前结构建议

```json
{
  "event_type": "plate_recognition",
  "timestamp": "2026-07-09T16:30:00.123Z",
  "device_id": "mobile_001",
  "bbox": [150, 400, 190, 420],
  "status": "normal",
  "data": {
    "plate_number": "京B6789T",
    "confidence": 0.83,
    "track_id": 3,
    "is_in_whitelist": false,
    "decision": "deny"
  }
}
```

#### 前端用途
- 最新车牌识别结果面板
- 识别记录
- 事件流
- 未来的视频车牌框叠加

> 说明：当前主运行链路 `main_server.py -> video_processor.py` 还未统一产出车牌识别数据，因此 `video_overlay.plates` 在当前主服务中可能为空；如果前端接的是独立车牌识别服务结果，则可以先用 `plate_recognition.bbox` 做叠加。

### 6.3 违停告警事件 `illegal_parking`

```json
{
  "event_type": "illegal_parking",
  "timestamp": "2026-07-09T16:30:05.123Z",
  "device_id": "mobile_001",

  "status": "warning",

    "track_id": 1,

    "zone_id": "no_parking_A",
    "zone_name": "禁停区A",
```

#### 前端用途

### 6.5 拥堵热力图事件 `traffic_density`

#### 后端当前会传

```json
{
  "event_type": "traffic_density",
  "timestamp": "2026-07-09T16:30:00.123Z",
  "device_id": "mobile_001",
  "status": "normal",
  "data": {
    "regions": [
      {
        "region_id": "road_A",
        "vehicle_count": 2,
        "status": "smooth",
        "color": "green"
      },
      {
        "region_id": "road_B",
        "vehicle_count": 6,
        "status": "congested",
        "color": "red"
      }
    ]
  }
}
```

#### 前端用途
- `TrafficHeatmap.vue` 这种独立热力图组件
- 区域统计卡片

#### 如果以后要把热力区也画回视频上
则建议后端额外扩展：

```json
{
  "region_id": "road_A",
  "vehicle_count": 2,
  "status": "smooth",
  "color": "green",
  "polygon": [[x1,y1],[x2,y2],[x3,y3],[x4,y4]]
}
```

这样前端既能做独立热力图，也能在视频上叠加区域轮廓。

## 8. 前端画框实现建议

### 8.1 统一以 `video_overlay` 为主

前端不要分别从零散事件里拼接当前帧框，而是优先使用：

```text
event_type = video_overlay
```

这样可以保证：

- 当前帧车辆框、违停框、异常框属于同一时刻；
- 前端知道什么时候清空旧框；
- 没检测到对象时，空数组即可表示“不画框”。

### 8.2 颜色建议

- `vehicles`：蓝色 / 橙色
- `plates`：青色 / 绿色
- `illegal_parking`：红色
- `road_anomalies`：紫色 / 红色

### 8.3 坐标缩放规则

后端传的是 **原始视频帧像素坐标**：

```text
[x1, y1, x2, y2]
```

前端在画布上绘制时，必须根据：

- `stream_size.width`
- `stream_size.height`
- 当前视频显示区域的实际宽高

做缩放映射后再画。

## 9. 后端控制台打印要求（必须保留）

为了方便后端人员监控，后端即使把 bbox 传给前端，也必须继续在控制台打印关键信息。

建议保留：

### 9.1 车辆检测摘要
```text
帧 60: 车辆检测 2 个，用于掩膜 2 个 [...]
```

### 9.2 车牌识别结果
```text
车牌识别: plate=京B6789T bbox=[...]
```

### 9.3 违停告警
```text
违停告警: track_id=1 stay_time=3.8s bbox=[...]
```

### 9.4 道路异常
```text
道路异常告警: bbox=[...] area=... duration=...
```

### 9.5 视频源状态
```text
成功连接视频流: ...
本地视频已读取结束: ...
视频流处理结束: ...
```

## 10. 给前端同学的任务清单

前端同学接下来要做的重点不是改后端，而是：

### 10.1 接收 `analysis_result`
按 `event_type` 分流处理：

- `vehicle_detection`
- `plate_recognition`
- `traffic_density`
- `illegal_parking`
- `road_anomaly`
- `video_overlay`

### 10.2 视频画框层优先接 `video_overlay`
- 收到 `video_overlay` 后清空旧图层
- 按 `no_parking_zones / vehicles / plates / illegal_parking / road_anomalies` 重绘当前帧

### 10.3 面板层继续接业务事件
- 车辆检测面板：读 `vehicle_detection`
- 车牌识别面板：读 `plate_recognition`
- 违停告警面板：读 `illegal_parking`
- 道路异常面板：读 `road_anomaly`
- 热力图：读 `traffic_density.data.regions`

### 10.4 拥堵分析
- 仍然以热力图组件形式呈现
- 如需后续叠加视频区域，可再利用区域 `polygon`

---

## 11. 一句话总结

当前后端已经具备统一事件推送能力，并已补充 `video_overlay` 作为按处理帧聚合的画框快照消息。当前前端已经接入该消息并能绘制车辆框、禁停区多边形、违停车框和道路异常框；车牌框字段已预留，等待车牌识别稳定并入主链路。

> 用 `analysis_result` 接收所有业务事件，用 `video_overlay` 专门做视频画框和区域叠加；后端继续保留控制台打印，前端负责把车辆框、禁停区、违停框、异常框真正画到视频画面上，而拥堵分析仍以热力图方式展示。
