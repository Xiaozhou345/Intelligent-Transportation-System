# 云边端协同智慧交通视觉感知系统系统设计报告

## 引言

### 1\.1 项目背景

随着城市交通规模不断扩大，交通场景中的车辆通行管理、道路拥堵监测、违停行为识别和异常物体检测等问题对实时感知能力提出了更高要求。传统交通监控系统通常依赖固定摄像头和人工巡检，存在部署成本较高、响应速度较慢、智能化程度不足等问题。

本项目拟模拟并开发一套“云边端协同”的智慧交通视觉感知系统。系统以移动端设备作为边端采集设备，以云端电脑作为 AI 分析服务器，以另一台电脑作为前端展示终端，实现从视频采集、视频传输、AI 分析到可视化展示的完整闭环。

### 1\.2 设计目标

本系统设计目标如下：

1. 实现移动端设备对交通场景视频的实时采集与传输；

2. 实现云端对视频流的接收、解码、抽帧和 AI 分析；

3. 支持车牌识别、车辆检测与拥堵热力图、违停跟踪告警、道路异常检测四类业务场景；

4. 实现云端分析结果向前端的实时推送；

5. 实现前端对视频画面、识别结果、拥堵状态和告警信息的可视化展示；

6. 保证系统在普通校园网络或热点环境下具备基本实时性、稳定性和可演示性。

### 1\.3 设计原则

系统设计遵循以下原则：

1. **三端解耦原则**：边端只负责采集和传输，云端负责 AI 分析，前端负责展示，降低模块之间的耦合度。

2. **云端集中分析原则**：车牌识别、车辆检测、目标跟踪和异常检测等计算密集型任务统一部署在云端，避免移动端设备算力不足。

3. **实时优先原则**：视频流处理采用低延迟传输、抽帧分析和异步推送策略，减少画面延迟和处理阻塞。

4. **可降级原则**：当实时推流不稳定时，系统支持视频段上传作为备用方案。

5. **可扩展原则**：系统模块之间通过统一接口交互，后续可扩展更多摄像设备、分析模型或交通场景。

## 系统总体架构设计

### 2\.1 总体架构

系统采用“边端采集层—云端分析层—前端展示层”的三层架构。

华为平板 / 手机边端
      ↓
实时视频采集、编码压缩、RTMP 推流
      ↓
云端电脑A
      ↓
视频流接收、解码、抽帧、AI 分析、业务判断
      ↓
WebSocket / HTTP API
      ↓
前端电脑B
      ↓
实时视频展示、识别结果展示、热力图展示、告警展示

### 2\.2 物理部署结构

系统物理部署分为三个部分：

考虑到边端平板与云端电脑A可能部署在不同楼层，系统不再限定三端设备必须处于同一 Wi\-Fi 环境，而是要求边端能够访问云端电脑A提供的 RTMP/HTTP 服务。系统优先测试校园网跨楼层直连；若校园网存在设备隔离或端口不可达，则采用虚拟局域网、内网穿透或公网中转方式建立视频传输通道，保证边端视频流能够稳定传输至云端。

## 技术选型设计

### 3\.1 边端技术选型

边端采用华为平板作为视频采集设备。第一阶段不自研完整 App，而是使用支持 RTMP 推流的摄像头推流软件完成视频采集与实时推流。这样可以降低移动端开发难度，优先保证系统整体链路可运行。

边端不部署核心 AI 模型，只进行轻量级视频处理，包括分辨率控制、帧率控制、码率控制、视频压缩、时间戳标记和设备编号标记。

### 3\.2 云端技术选型

云端部署在电脑A上，负责接收视频流并完成 AI 分析。

### 3\.3 前端技术选型

前端部署在电脑B上，负责展示实时视频、分析结果和告警信息。

## 边端采集与传输模块设计

### 4\.1 模块职责

边端采集与传输模块由成员A负责，核心目标是解决“视频如何从平板稳定、低延迟地传输到云端”的问题。

本模块主要职责包括：

1. 使用华为平板摄像头采集交通场景视频；

2. 对视频进行 H\.264 编码压缩；

3. 控制视频分辨率、帧率和码率；

4. 通过 RTMP 将实时视频流推送至云端；

5. 在推流失败时支持 HTTP 视频段上传；

6. 向云端提供设备编号、视频流地址、时间戳等信息；

7. 保证云端 AI 分析模块能够稳定读取视频帧。

### 4\.2 边端运行方式

本项目第一版采用“固定平板 \+ 推流软件”的方式实现边端采集。

具体流程如下：

1\. 将华为平板固定在支架上；
2\. 平板摄像头对准学院704智慧交通沙盘或道路区域；
3\. 平板连接至可访问云端电脑A的网络环境；
4\. 打开支持 RTMP 的推流软件；
5\. 输入云端 RTMP 推流地址；
6\. 设置分辨率、帧率、码率和编码格式；
7\. 点击开始推流；
8\. 云端电脑A接收并解码视频流。



该方式不要求第一阶段开发完整移动端 App，能够快速完成边端到云端的视频链路验证。

边端默认视频参数设计如下：

### 4\.4 边端推流地址设计

RTMP 推流地址格式设计如下：


云端 AI 模块通过该视频流地址拉取视频流，并从视频流中读取图像帧。

### 4\.5 设备注册信息设计

边端开始推流后，需要向云端提供设备注册信息。设备注册信息采用 JSON 格式。

```JSON
{
  "device_id": "mobile_001",
  "device_type": "huawei_tablet",
  "stream_url": "rtmp://192.168.1.100:1935/live/mobile_001",
  "scene_id": "scene_704_sandbox",
  "resolution": "1280x720",
  "fps": 15,
  "codec": "H.264",
  "bitrate": "2Mbps",
  "timestamp": "2026-07-06 13:00:00",
  "status": "online"
}
```

云端接收到设备注册信息后，将该设备加入视频流管理列表，并根据 `stream_url` 拉取视频流。

### 4\.6 边端轻量级预处理设计

边端不进行车牌识别、车辆检测、车辆跟踪和道路异常检测等核心 AI 推理。边端仅进行如下轻量级预处理：

1. **视频压缩**：通过 H\.264 编码降低传输带宽；

2. **帧率控制**：将帧率控制在 15fps 左右，避免云端处理压力过大；

3. **分辨率控制**：根据网络情况选择 1080p、720p 或 480p；

4. **码率控制**：根据 Wi\-Fi 或热点稳定性调节码率；

5. **时间戳标记**：为视频流或设备信息添加采集时间；

6. **设备编号标记**：标识当前视频来源；

7. **可选区域裁剪**：在固定视角下，只保留道路或沙盘区域，减少无关背景。

### 4\.7 断流与备用机制设计

为保证系统稳定运行，边端设计如下容错机制：

1. **断流提示**：当推流软件或云端检测到视频流中断时，系统提示设备离线；

2. **手动重连**：用户重新点击推流按钮，恢复视频流；

3. **参数降级**：网络较差时降低分辨率、帧率或码率；

4. **备用上传**：若实时推流持续失败，平板可录制短视频段，并通过 HTTP 上传至云端；

5. **云端最新帧策略**：云端处理不过来时只保留最新帧，丢弃过旧帧，避免延迟累积。

## 云端 AI 分析模块设计

### 5\.1 模块职责

云端 AI 分析模块由成员B负责，主要职责包括：

1. 接收边端传来的视频流或视频段；

2. 对视频进行解码和抽帧；

3. 根据业务场景调用对应 AI 模型；

4. 完成车牌识别、车辆检测、违停跟踪和异常检测；

5. 执行业务逻辑判断；

6. 将分析结果推送至前端。

### 5\.2 视频接收与取帧流程

云端接收到边端 RTMP 视频流后，使用 FFmpeg 或 OpenCV 进行解码取帧。

RTMP 视频流
   ↓
FFmpeg / OpenCV 解码
   ↓
获取图像帧
   ↓
抽帧策略
   ↓
AI 模型推理
   ↓
业务逻辑处理
   ↓
结果推送前端

为避免云端处理压力过大，系统不要求每一帧都进入 AI 模型。推荐采用抽帧策略，例如：

视频输入：15fps
AI 分析：每 3 帧处理 1 帧
实际推理：约 5fps

未进行深度学习推理的帧可用于视频显示或由跟踪算法进行轨迹预测。

### 5\.3 车牌识别与通行决策设计

车牌识别模块主要用于识别视频画面中的车辆牌照，并与本地白名单车辆库进行比对，输出是否允许通行的决策结果。

处理流程如下：

视频帧输入
   ↓
车辆/车牌区域检测
   ↓
车牌图像裁剪
   ↓
LPRNet 字符识别
   ↓
输出车牌号
   ↓
查询本地白名单
   ↓
生成通行决策

输出结果示例：

```JSON
{
  "event_type": "plate_recognition",
  "timestamp": "2026-07-06 13:00:01",
  "device_id": "mobile_001",
  "plate_number": "京A12345",
  "is_in_whitelist": true,
  "decision": "allow",
  "bbox": [120, 80, 220, 140]
}
```

### 5\.4 车辆检测与拥堵热力图设计

车辆检测模块用于检测画面中所有车辆目标，并根据车辆所在区域统计车辆密度，生成拥堵状态。

处理流程如下：

视频帧输入
   ↓
YOLO 车辆检测
   ↓
获取车辆 Bounding Box
   ↓
根据区域划分统计车辆数量
   ↓
计算区域拥堵等级
   ↓
生成热力图数据
   ↓
推送前端展示

区域拥堵等级可设计为：

输出结果示例：

```JSON
{
  "event_type": "traffic_density",
  "timestamp": "2026-07-06 13:00:02",
  "device_id": "mobile_001",
  "regions": [
    {
      "region_id": "road_A",
      "vehicle_count": 2,
      "status": "smooth",
      "color": "green"
    },
    {
      "region_id": "road_B",
      "vehicle_count": 6,
      "status": "congested",
      "color": "red"
    }
  ]
}
```

### 5\.5 违停跟踪与告警设计

违停检测不能只依赖单帧目标检测，而需要对进入禁停区域的车辆进行持续跟踪，计算其停留时长。

处理流程如下：

视频帧输入
   ↓
YOLO 检测车辆
   ↓
ByteTrack 分配车辆 ID
   ↓
判断车辆是否进入禁停区域
   ↓
记录进入时间
   ↓
持续更新车辆位置
   ↓
计算停留时长
   ↓
超过阈值后触发告警

禁停区域可由前端或配置文件预先设定，例如使用多边形坐标表示：

```JSON
{
  "no_parking_zone": [
    [100, 200],
    [500, 200],
    [500, 400],
    [100, 400]
  ]
}
```

违停告警输出示例：

```JSON
{
  "event_type": "illegal_parking",
  "timestamp": "2026-07-06 13:00:10",
  "device_id": "mobile_001",
  "track_id": 17,
  "stay_time": 32.5,
  "threshold": 30,
  "status": "warning",
  "bbox": [230, 180, 310, 260]
}
```

### 5\.6 道路异常检测设计

道路异常检测面向未知目标，例如掉落物、纸箱、石块等。这类目标无法完全通过固定类别的监督目标检测模型识别，因此采用“背景建模 \+ 动态掩膜”的思路。

处理流程如下：

视频帧输入
   ↓
MOG2 背景建模
   ↓
提取前景变化区域
   ↓
YOLO 检测正常车辆
   ↓
将车辆区域从前景图中剔除
   ↓
分析剩余前景连通域
   ↓
判断是否为静止异常物体
   ↓
触发告警并标注位置

异常判定条件可设计为：

1. 连通域面积大于设定阈值；

2. 连续 N 帧存在；

3. 位置变化较小；

4. 不属于正常车辆检测框区域；

5. 位于道路区域内。

异常告警输出示例：

```JSON
{
  "event_type": "road_anomaly",
  "timestamp": "2026-07-06 13:00:15",
  "device_id": "mobile_001",
  "anomaly_type": "unknown_object",
  "status": "warning",
  "affected_lane": "lane_1",
  "bbox": [360, 220, 420, 280],
  "duration_frames": 20
}
```

前端展示模块由成员C负责，主要用于展示云端分析结果，帮助用户直观理解当前交通状态。

主要功能包括：

## 接口设计

### 7\.1 边端到云端视频流接口

边端通过 RTMP 将视频流推送到云端。

协议：RTMP
地址格式：rtmp://\<server\_ip\>:1935/live/\<device\_id\>
编码格式：H\.264
推荐分辨率：1280×720
推荐帧率：15fps
推荐码率：1\.5Mbps—3Mbps

### 7\.2 边端设备注册接口

接口功能：边端开始推流时向云端登记设备与视频流信息。

请求方式：POST
接口地址：/api/edge/register
数据格式：JSON

请求示例：

```JSON
{
  "device_id": "mobile_001",
  "device_type": "huawei_tablet",
  "stream_url": "rtmp://192.168.1.100:1935/live/mobile_001",
  "scene_id": "scene_704_sandbox",
  "resolution": "1280x720",
  "fps": 15,
  "codec": "H.264",
  "bitrate": "2Mbps",
  "timestamp": "2026-07-06 13:00:00"
}
```

响应示例：

```JSON
{
  "code": 200,
  "message": "device registered successfully",
  "device_id": "mobile_001",
  "status": "online"
}
```

### 7\.3 视频段上传接口

接口功能：当实时推流不可用时，边端上传短视频段供云端分析。

请求方式：POST
接口地址：/api/video/upload
数据格式：multipart/form\-data

上传字段设计：

响应示例：

```JSON
{
  "code": 200,
  "message": "video uploaded successfully",
  "task_id": "task_20260706_001",
  "status": "pending"
}
```

### 7\.4 云端到前端结果推送接口

云端通过 WebSocket 向前端推送分析结果。

协议：WebSocket
地址：ws://\<server\_ip\>:\<port\>/ws/results
数据格式：JSON

统一结果格式：

```JSON
{
  "event_type": "plate_recognition",
  "timestamp": "2026-07-06 13:00:01",
  "device_id": "mobile_001",
  "data": {},
  "bbox": [],
  "status": "normal"
```

## 实时性与性能优化设计

### 10\.1 抽帧分析策略

由于视频流可能达到 15fps 或 30fps，如果每一帧都输入 AI 模型，会导致推理压力过大。因此系统采用抽帧分析策略。

示例：

第 1 帧：进行 AI 推理
第 2 帧：跳过或用于显示
第 3 帧：跳过或用于显示
第 4 帧：进行 AI 推理

该策略可以降低云端推理负载，同时保证前端仍然能够看到相对连续的视频画面。

### 10\.2 最新帧缓存策略

云端视频处理模块采用最新帧缓存策略：

1\. 视频接收线程持续读取最新帧；
2\. AI 分析线程只读取当前最新帧；
3\. 如果 AI 处理较慢，旧帧直接丢弃；
4\. 避免视频帧在队列中长期积压。

这样可以避免系统延迟越来越大。

### 10\.3 异步处理策略

车牌 OCR、异常检测、告警记录写入等任务可以采用异步处理方式，避免阻塞主视频流。

视频解码线程
   ↓
最新帧缓存
   ↓
AI 推理线程
   ↓
业务逻辑线程
   ↓
WebSocket 推送线程

## 部署设计

### 11\.1 网络部署

系统支持边端平板、云端电脑A和前端电脑B部署在不同物理位置。网络部署的核心要求是：边端平板能够访问云端电脑A提供的 RTMP 推流服务和 HTTP 接口，前端电脑B能够访问云端电脑A提供的 Web 服务和 WebSocket 服务。

考虑到边端平板与云端电脑A可能位于不同楼层，系统不再限定三端设备必须处于同一局域网中，而是要求三端之间具备网络可达性。若校园网允许跨设备访问，可直接使用云端电脑A在校园网中的 IP 地址进行通信；若校园网存在设备隔离或端口不可达，则采用虚拟局域网、内网穿透或公网中转方式建立通信链路。

边端推流地址格式如下：

```Plain Text
rtmp://<云端可访问地址>:1935/live/<device_id>
```

示例：

```Plain Text
rtmp://<云端可访问地址>:1935/live/mobile_001
```

前端访问地址格式如下：

```Plain Text
http://<云端可访问地址>:5000
```

WebSocket 地址格式如下：

```Plain Text
ws://<云端可访问地址>:5000/ws/results
```

其中，`<云端可访问地址>` 可以是校园网 IP、虚拟局域网 IP、内网穿透地址或公网中转地址。系统实际部署时，应先测试边端平板是否能够访问云端电脑A的 RTMP 端口和 HTTP 服务端口，确认网络可达后再进行视频推流和系统联调。

## 总结

本系统采用云边端协同架构，以华为平板作为边端采集设备，以云端电脑作为 AI 分析服务器，以前端电脑作为可视化展示终端。边端主要负责视频采集、编码压缩和 RTMP 推流，云端负责视频流接收、取帧、AI 推理和业务逻辑判断，前端负责实时展示视频画面、识别结果、拥堵热力图和告警信息。

在具体实现上，系统以 RTMP 实时推流作为主传输方案，以 HTTP 视频段上传作为备用方案；以 YOLO、LPRNet、ByteTrack 和 MOG2 等方法完成不同交通视觉任务；以 WebSocket 实现云端分析结果到前端的实时推送。该设计能够较好地体现“边端采集、云端分析、前端展示”的协同思想，并具备较强的可实现性和可演示性。
