# 云边端协同智慧交通视觉感知系统需求分析文档

## 1\. 项目概述

本项目旨在模拟并开发一套“云边端协同”的智慧交通视觉感知系统。系统采用轻量化边端采集、云端高算力分析、独立前端展示的物理分离架构，主要应用于智慧交通沙盘或真实道路场景。

**物理部署拓扑：**

**边端（手机）：** 负责实时视频流/视频段采集与网络推流，可视情况承担轻量级预处理。

**云端（电脑A）：** 接收视频流，提供强大的AI算力，运行多模型智能调度与业务逻辑判定。

**前端（电脑B）：** 接收云端分析结果，进行大屏可视化渲染与告警展示。

## 2\. 功能需求分析 \(业务场景\)

系统围绕智慧交通管理，核心涵盖四大AI视觉业务场景：

|业务场景|功能描述|核心约束与技术方向|
|---|---|---|
|**车牌识别与通行决策**|实时检测画面中的车辆并识别车牌，与本地白名单库比对，输出道闸开放决策。|需保证高准确率（OCR\+检测模型）；无需实际控制硬件道闸。<br>技术选型：YOLOv11s/m \(目标检测\) \+ LPRNet \(字符识别\)|
|**车辆检测与热力图**|检测画面中所有车辆目标，统计区域车辆密度，动态生成拥堵热力图；系统拓扑图需直观显示红绿（拥堵）状态。|关注全局视野的密度统计；采用目标检测模型（如YOLO系列）。<br>技术选型**：** YOLOv11s/m \(COCO预训练\) \+ OpenCV 矩阵运算|
|**违停跟踪与告警**|对进入禁停区域的单一车辆进行持续**跟踪**（非单纯检测），计算停留时长，超时触发告警。|必须采用目标跟踪算法，确保车辆跨帧的唯一身份关联。<br>技术选型**：**YOLOv11s/m \+ ByteTrack|
|**道路异常检测**|检测路面上本不应存在的异常物体，即时触发告警并标注受影响车道及具置。|**目标未知**，无法使用常规监督学习目标检测。需考虑背景建模、无监督异常检测等方案。<br>技术选型：MOG2 \(混合高斯背景建模\) \+ YOLO 动态掩膜|

## 3\. 系统架构与模块需求 \(依据角色划分\)

### 3\.1 边端采集与传输层 \(成员A\)

**核心目标：解决“视频如何从手机高效稳定传输到云端”**

**数据采集需求：**
边端应支持使用手机或平板等移动端设备作为视频采集设备，调用设备摄像头获取学院704智慧交通沙盘或道路场景下的实时视频画面。采集内容应覆盖车辆通行区域、道路区域、禁停区域以及可能出现异常物体的路面区域，为云端 AI 分析模块提供连续、清晰的视频输入。

**视频传输需求：**
边端应支持将采集到的视频数据稳定传输至云端电脑A，保证云端能够实时接收视频流或视频段并进行后续 AI 分析。系统优先考虑采用 RTMP 推流方式实现实时视频传输，以满足低延迟和连续采集需求；同时保留 HTTP 视频段上传作为备用方案，用于网络环境不稳定或实时推流失败时的分段分析。WebSocket 主要用于控制信令或状态信息传输，不作为主要视频传输方式。

**视频参数控制需求：**
边端应支持对视频分辨率、帧率、码率和编码格式进行配置，以适应不同网络环境和云端处理能力。视频流应支持 H\.264 等常见编码格式，初步目标参数为 1280×720 分辨率、15fps 帧率，并可根据网络带宽和系统负载调整为更高或更低的视频质量。通过参数控制，在视频清晰度、实时性和传输稳定性之间取得平衡。

**轻量级预处理需求：**
边端不承担车牌识别、车辆检测、车辆跟踪和道路异常检测等核心 AI 推理任务，只进行必要的轻量级预处理。预处理内容包括视频压缩、帧率控制、分辨率调整、时间戳标记、设备编号标记等，用于降低网络带宽占用并提高云端接收效率。在设备性能允许的情况下，可支持简单的画面变化检测或道路区域裁剪，但相关功能不作为系统核心 AI 分析能力。

**设备信息上报需求：**
边端在开始传输视频时，应向云端提供必要的设备与视频流信息，包括设备编号、视频流地址、分辨率、帧率、编码格式、场景编号和时间戳等。云端 AI 分析引擎可根据这些信息完成视频流接入、设备区分和后续分析任务调度。

**传输稳定性需求：**
边端应具备基本的传输稳定性保障能力。当移动端与云端之间的网络连接异常或视频流中断时，系统应能够提示连接异常，并支持重新连接或切换至备用视频段上传方式。云端处理压力较大时，边端应支持通过降低帧率、分辨率或码率的方式减轻传输压力，保证系统整体可用性。



### 3\.2 云端AI分析引擎 \(成员B\)

**核心目标：解决“如何高效处理视频并实现多场景AI分析”**

**流媒体接收：** 搭建稳定服务，接收来自边端的实时视频流或分段视频。

**多模型动态切换机制（核心难点）：** 系统必须具备多模型调度能力。根据当前业务场景需求，智能加载、卸载或切换对应的AI模型，避免显存溢出。

**业务逻辑处理：** 实现白名单比对、区域驻留时长计算、异常判定与告警触发逻辑。

**性能调优与负载均衡：**

**跳帧处理：** 针对模型单帧推理耗时导致无法实时处理的问题，设计智能跳帧逻辑（如处理第1帧，丢弃2、3帧，处理第4帧）。

**分段分析：** 支持将视频切片后进行异步分段分析，保障系统的顺畅丝滑。

### 3\.3 前端展示与系统集成 \(成员C\)

**核心目标：解决“如何直观展示数据并保障整体可用性”**

**实时视频渲染：** 流畅播放云端处理后或直拉的实时视频流。

**多维数据可视化：**** **

实时展示车牌识别结果与白名单状态。

动态绘制拥堵热力图及红绿状态拓扑图。

提供违停告警列表（含时长显示）与异常物体位置标注。

**通信与后台管理：** 通过 WebSocket 接收云端高频推送结果；提供简单的后台配置面板（白名单管理、超时阈值配置）。

## 4\. 接口与通信需求

为确保三端协同，需严格定义以下交互接口：

**边端 → 云端（A → B）：**

**视频流规约：** 边端应支持向云端传输实时视频流或视频段文件。系统优先支持 RTMP 推流作为实时视频传输方式，并保留 HTTP 视频段上传作为备用方式。视频流应支持 H\.264 等常见编码格式，并支持分辨率、帧率和码率配置。初步约定推荐参数为 1280×720、15fps、1\.5Mbps—3Mbps，后续可根据网络环境和云端处理能力进行调整。

**设备注册信息：** 边端开始传输视频时，应向云端提供设备编号、视频流地址、分辨率、帧率、编码格式、场景编号和时间戳等信息，便于云端进行视频流管理、设备区分和任务调度。

**备用上传机制：** 当实时推流受网络环境影响无法稳定运行时，边端应支持将视频切分为短视频段，并通过 HTTP 接口上传至云端，由云端进行分段分析。



**云端 → 前端 \(B → C\)：**

**数据结构定义：** 统一采用 JSON 格式传递分析结果。包含：时间戳、事件类型（识别/拥堵/违停/异常）、坐标信息（Bounding Box）、业务数据（车牌号/时长等）。

**推送频率控制：** 需约定推送频率，避免前端渲染过载。

**前端 ↔ 云端 \(C ↔ B\)：**

**双向控制信令：** 采用 WebSocket 实现双向通信。前端可向云端下发“切换场景/模型”指令，云端向前端实时推送告警。

## 5\. 核心技术难点与应对策略

**高并发与实时性的矛盾 \(顺畅丝滑保障\)：**

**挑战：** 视频流高达 30 FPS，如果每一帧都强行送入深度学习模型，单帧数十毫秒的推理耗时叠加，必然导致画面延迟卡顿或内存队列积压崩溃。

**策略（智能抽帧 \+ 异步流水线）：**

- 引入**智能跳帧管道**：深度学习网络（YOLO）无需逐帧检测，可设定跳帧率（如每 3 帧送入 1 帧）。

- 结合**卡尔曼滤波预测**：在未进行深度学习推理的空档帧，利用 ByteTrack 底层的卡尔曼滤波直接预测车辆轨迹，既降低了 60% 的算力压力，又补全了视觉帧率，实现真正的 30 FPS 丝滑渲染。

- **异步解耦**：车牌 OCR 识别等局部长耗时任务，放入后台子线程或异步队列处理，绝不阻塞主视频流的渲染线程。

**多模型共存与动态调度：**

**挑战：** 4 个业务场景涉及目标检测、多目标跟踪、OCR、背景建模等多种算法，如果资源分配不当，容易抢占算力导致系统崩溃。

**策略（GPU/CPU异构计算 \+ 模型并行常驻）：**

- 将最吃算力的 YOLOv11 和 LPRNet 一次性加载并常驻 GPU 显存，彻底消除模型切换时的 I/O 读取延迟，实现场景间的零秒级无缝切换。

- 将 ByteTrack（跟踪运算）、MOG2（背景建模）及业务比对逻辑下放至 CPU 运行，实现 GPU 与 CPU 的完美异构负载均衡。

**未知目标的异常检测：**

**挑战：** 道路异常掉落物（石头、纸箱等）无固定类别且无法穷举，常规基于监督学习的 YOLO 模型无法识别“未知目标”。

**策略（MOG2 背景建模 \+ YOLO 动态掩膜）：**

- 转换思路，不识别“是什么”，只捕捉“不该出现的变动”。敲定采用 OpenCV 的 **MOG2 混合高斯算法**进行背景建模，提取所有异常前景。

- 引入**动态交通流掩膜**：利用 YOLOv11 识别出的正常车辆 Bounding Box，在 MOG2 的前景图中将其涂黑（消除正常车辆的干扰）。

- 对剩余的前景连通域进行时空追踪，若某连通域面积大于阈值且连续 N 帧在画面中保持静止，则确认为异常掉落物并告警。
