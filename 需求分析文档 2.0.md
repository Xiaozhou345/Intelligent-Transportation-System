# 1 项目概述

## 1\.1 项目背景

传统交通监控依靠固定摄像头\+人工巡检模式，存在部署成本高、智能化程度低、响应滞后等缺陷，无法自动完成车牌识别、拥堵统计、违停判定、路面异物告警等工作。

本项目搭建**云边端协同智慧交通视觉感知系统**，分离采集、AI算力、可视化三层硬件，采用移动端平板作为边缘采集设备、GPU电脑作为云端AI服务器、PC作为前端监控大屏，构建视频采集\-网络传输\-智能分析\-可视化告警完整业务闭环，适用于校园交通沙盘、小型园区道路演示场景。

## 1\.2 系统部署拓扑

系统分为三层硬件，三设备可跨校园网、热点部署，无需同一局域网，媒体 SRT/RTMP 端口平板直接访问云端，仅 5000 端口 HTTP/WebSocket 信令使用 FRP 内网穿透，FRP 不转发视频大流：

1. **边端（华为平板/手机）**：仅负责视频实时采集、编码压缩、网络推流，不运行AI模型，降低边缘设备算力门槛；

2. **云端（GPU电脑）**：集中部署多套深度学习模型，承担车辆检测、车牌识别、多目标跟踪、背景建模等全部算力任务；

3. **前端（PC电脑）**：接收云端实时分析数据，完成视频渲染、检测框叠加、拥堵热力图、告警列表可视化展示。

## 1\.3 核心技术特点

1. **云边端解耦协同**：采集、算力、展示三层完全解耦，采用标准化 SRT/RTMP、HTTP、WebSocket 接口通信；

2. **多模型动态调度**：云端按需加载/切换YOLO、LPRNet、ByteTrack、MOG2多套视觉模型，避免显存溢出；

3. **低延迟实时链路**：采用抽帧推理、异步流水线、卡尔曼轨迹补偿降低画面延迟；

4. **双传输容错机制**：SRT 实时推流为主、RTMP 为兼容备用链路；

5. **无监督异常检测**：不依赖异物样本，通过背景建模\+车辆掩膜识别道路未知掉落物。

## 1\.4 项目整体目标

1. 实现移动端设备摄像头实时采集交通画面，以 SRT 为主通道低延迟推流，RTMP 作为兼容备用推流；

2. 云端完成视频流解码、智能抽帧、多模型AI推理与业务逻辑判定；

3. 支撑四大核心交通业务：车牌识别白名单通行、区域拥堵热力统计、违停跟踪告警、道路异物检测；

4. 云端通过WebSocket向前端实时推送结构化JSON分析结果；

5. 前端实现视频、检测框、热力图、告警信息一体化可视化监控；

6. 在校园WiFi、手机热点普通网络环境下，保证系统稳定可完整演示。

# 2 功能需求分析

## 2\.1 车牌识别与通行决策模块

|功能项|功能描述|技术实现约束|
|---|---|---|
|车辆与车牌区域检测|从视频帧中定位所有车辆，裁剪车牌感兴趣区域|YOLOv11轻量化检测模型，可适配沙盘小车小车牌|
|端到端车牌字符识别|识别国内31省简称、数字、大小写字母、港/澳/警特殊字符|LPRNet \+ CTC损失函数，字符总数68类|
|白名单比对判定|将识别出的车牌与本地车辆白名单库匹配，输出通行决策|MySQL本地白名单数据表，支持增删改查|
|识别事件持久化|记录每一次识别时间、设备、车牌、放行状态|每条识别记录入库，支持历史查询|
|实时结果推送|识别完成立即向前端推送车牌、坐标、通行结果|WebSocket JSON实时推送|

性能需求：车牌整体识别准确率≥95%，单帧推理耗时≤100ms。

## 2\.2 车辆检测与拥堵热力图模块

|功能项|功能描述|技术实现约束|
|---|---|---|
|全画面车辆检测|识别画面内所有机动车，输出包围框坐标|YOLOv11s，适配沙盘小型车辆|
|自定义道路区域划分|支持前端绘制多边形道路分区，作为统计单元|多边形坐标配置持久化存储|
|区域车辆计数|统计每个道路分区内实时车辆数量|基于检测框中心坐标归属判定|
|拥堵等级分级|根据车辆数量划分畅通、缓行、拥堵三档|可自定义数量阈值|
|热力数据生成|将拥堵等级映射绿/黄/红三色热力数据，推送前端绘图|每秒至少刷新一次热力数据|

拥堵分级规则：

1. 畅通（绿色）：区域车辆≤2辆；

2. 缓行（黄色）：区域车辆3\~5辆；

3. 拥堵（红色）：区域车辆＞5辆。

## 2\.3 车辆跟踪与违停告警模块

|功能项|功能描述|技术实现约束|
|---|---|---|
|多目标持续跟踪|给画面每辆车分配唯一track\_id，跨帧关联同一车辆|ByteTrack算法，卡尔曼滤波预测|
|禁停区域配置|前端绘制多边形禁停区域，后台持久保存坐标|支持多块禁停区域同时生效|
|停留时长计时|车辆进入禁停区开始计时，离开清零|精确到0\.1秒时间戳统计|
|超时违停告警|车辆停留超过预设阈值，生成告警消息|阈值可配置（5\~600秒）|
|告警记录存储|保存违停车辆ID、停留时长、画面坐标、告警时间|告警数据支持历史查询导出|

## 2\.4 道路异常物体检测模块

### 业务痛点

道路掉落纸箱、石块等异物无固定类别，无法使用常规目标检测训练样本识别，采用背景建模方案。

|功能项|功能描述|技术实现约束|
|---|---|---|
|道路可通行区域分割|区分路面与非道路区域，仅在路面检测异物|道路分割模型|
|动态车辆掩膜剔除|使用YOLO检测框涂黑车辆区域，消除车辆运动干扰|推理帧实时生成掩膜|
|MOG2背景建模|提取画面静态前景物体作为异常候选|固定机位场景核心方案|
|多时序校验|异物连续多帧存在、位置变化极小才判定告警|过滤光影、树叶抖动误报|
|异常信息推送|推送异物坐标、影响车道、持续帧数至前端告警|异常弹窗高亮标注区域|

异常判定硬性条件：

1. 连通域面积大于设定阈值；

2. 连续N帧稳定存在无大幅位移；

3. 区域属于可行驶路面，排除花草、边界干扰；

4. 不属于任意车辆检测框覆盖范围。

## 2\.5 边端采集传输模块

核心目标：稳定、低带宽将视频流从平板传输至云端，不运行AI推理。

1. 视频采集：调用平板原生摄像头，支持720P/1080P/480P分辨率切换；

2. 编码压缩：统一H\.264编码，可手动调节码率平衡画质与带宽；

3. SRT 实时推流（主推流）：主传输方案，地址规范`srt://云端IP:8890?streamid=publish:live/{device_id}`；RTMP 为兼容备用推流，地址规范`rtmp://云端IP:1935/live/{device_id}`

4. 设备注册上报：平板启动推流后，HTTP上传设备型号、流地址、分辨率等信息；

5. 心跳在线检测：周期性上报在线状态，云端判定设备离线；

6. 备用分段上传：RTMP断流时，自动截取短视频HTTP上传云端做离线分析；

7. 轻量预处理：仅帧率、分辨率裁剪，无任何深度学习计算；

8. 网络降级：WiFi卡顿自动降低帧率、码率保障连通。

## 2\.6 云端流媒体与调度模块

1. 多路流接入转发：部署 MediaMTX 流媒体服务，同时监听 8890 \(UDP SRT\)、1935 \(TCP RTMP\) 接收平板推流，完成协议转封装，分流为 WHEP/WebRTC 前端播放通道、RTSP 8554 AI 拉流通道；

2. RTSP 独立子进程解码：由单独采集子进程通过 RTSP 拉流，使用 FFmpeg\+OpenCV 完成视频解码，生成图像矩阵；MediaMTX 仅负责协议转发，不执行解码操作。

3. 智能抽帧：15fps视频无需逐帧推理，间隔多帧送入AI降低算力；

4. 最新帧缓存：推理阻塞时丢弃旧帧，杜绝延迟持续累积；

5. 多模型调度：按需加载/释放YOLO、LPRNet、ByteTrack，控制显存占用；

6. 异步任务队列：识别、告警入库等耗时任务异步执行，不阻塞视频主线程；

7. 统一消息封装：所有AI结果标准化JSON，通过WebSocket推送前端。

## 2\.7 前端可视化监控模块

前端采用 WHEP/WebRTC 拉取云端实时视频作为主播放链路，WebRTC 异常时自动降级 HLS 播放，页面大屏布局划分：

1. 顶部状态栏：系统标题、当前时间、所有设备在线状态；

2. 左侧主画面：实时视频流，Canvas叠加车辆框、车牌框、禁停区、异常框；

3. 右侧信息面板：实时车牌列表、通行决策、违停告警、道路异常告警；

4. 底部热力面板：分区域红绿黄拥堵热力图实时渲染。

功能需求：

1. WebSocket长连接实时接收云端分析数据流；

2. 画面叠加图形自定义颜色、粗细；

3. 支持手动绘制道路、禁停多边形并下发云端保存；

4. 可切换四大业务分析模式（车牌/拥堵/违停/异物）；

5. 告警列表实时滚动，支持历史告警分页查询；

6. 白名单增删改查可视化表单；

7. 检测阈值、违停时长参数前端下发云端生效。

# 3 系统分层需求

## 3\.1 边端采集层需求

### 3\.1\.1 采集需求

- 支持华为安卓平板、手机摄像头实时采集；

- 画面支持720P默认输出，可降级480P；

- 帧率固定15fps，可下调至10/5fps适配弱网。

### 3\.1\.2 传输需求

- 主通道 SRT 低延迟推流，端口 8890 \(UDP\)；RTMP 为 TCP 1935 兼容备用通道；

- 备用HTTP短视频上传，断流自动触发；

- 设备启动主动注册，每30秒发送心跳包。

### 3\.1\.3 预处理约束

禁止在边缘设备部署任何检测、识别AI模型，仅做编码缩放。

### 3\.1\.4 容错需求

- 推流断开弹窗提示，一键重连；

- 网络极差自动降低码率、分辨率；

- 本地缓存短视频用于备用上传。

## 3\.2 云端AI分析层需求

### 3\.2\.1 流媒体需求

- 稳定接收多路平板RTMP流，支持多设备并行；

- 实时解码、帧缓存、智能跳帧控制负载。

### 3\.2\.2 AI算法需求

- YOLOv11车辆/车牌双检测；

- LPRNet端到端车牌字符识别；

- ByteTrack多目标跟踪；

- MOG2\+道路分割异物检测；

- 多模型共存，按需加载释放显存。

### 3\.2\.3 业务逻辑需求

白名单比对、拥堵计数、违停计时、异物过滤、事件入库。

### 3\.2\.4 通信需求

- WebSocket长连接向前端推送结构化JSON结果；

- HTTP接口接收边端注册、前端配置指令。

## 3\.3 前端展示层需求

### 3\.3\.1 可视化需求

实时视频、检测框、热力图、告警列表同步渲染。

### 3\.3\.2 交互需求

场景切换、区域绘制、阈值配置、历史查询、白名单管理。

### 3\.3\.3 通信需求

WebSocket双向通信，既能接收分析数据，也能下发控制指令至云端。

# 4 接口通信需求

## 4\.1 边端→云端通信接口

1. SRT 主推流、RTMP 备用视频流接口

协议：主推 SRT \(UDP 8890\)，地址规范`srt://云端IP:8890?streamid=publish:live/{device_id}`；

备用 RTMP \(TCP 1935\)，地址规范`rtmp://云端IP:1935/live/{device_id}`；编码统一 H\.264。

2. 设备注册接口

请求方式POST；地址`/api/edge/register`；JSON上报设备全量信息；

3. 心跳上报接口

POST，周期30秒，上报设备在线状态；

4. 视频段备用上传接口

POST form\-data，上传短视频文件做离线分析。

## 4\.2 云端→前端推送接口

WebSocket 仅传输 AI 结构化 JSON 告警数据，实时视频由前端独立拉取 WebRTC/HLS 流，两条链路相互隔离。通信协议：WebSocket；地址`ws://云端IP:5000/ws/results`

统一JSON推送格式：

```JSON
{
  "event_type": "plate_recognition/illegal_parking/traffic_density/road_anomaly",
  "timestamp": "2026-07-10 16:20:30",
  "device_id": "mobile_001",
  "data": {},
  "bbox": [],
  "status": "normal/warning"
}
```

## 4\.3 前端→云端控制接口

基于WebSocket下发控制JSON，支持指令：

1. switch\_scene：切换AI分析业务场景；

2. set\_threshold：修改违停、异物判定阈值；

3. draw\_zone：下发道路/禁停多边形坐标；

4. reload\_whitelist：重载白名单；

5. reset\_anomaly\_bg：重置背景建模。

# 5 核心技术难点与解决方案

## 5\.1 实时性与算力冲突问题

### 痛点

15fps视频逐帧执行YOLO\+LPRNet推理，GPU算力不足，画面延迟持续堆积。

### 解决策略

1. 智能抽帧：每3帧仅1帧完整AI推理，中间帧使用ByteTrack卡尔曼预测轨迹；

2. 异步多线程解耦：解码、推理、推送三条独立线程互不阻塞；

3. 旧帧丢弃机制：推理慢时直接抛弃过期缓存帧，延迟不累积。

## 5\.2 多模型共存显存调度问题

### 痛点

同时加载YOLO、LPRNet、ByteTrack、分割模型显存溢出，多场景切换卡顿。

### 解决策略

1. 高频YOLO、LPRNet常驻GPU；

2. 异物分割、MOG2低频模型使用时加载、闲置释放；

3. 推理输入分辨率动态缩小降低显存占用。

## 5\.3 未知道路异物检测难题

### 痛点

掉落物无固定类别，无法用常规监督学习训练，车辆、光影极易产生误报。

### 解决策略

1. 道路分割模型限定检测区域仅路面；

2. YOLO检测框生成掩膜，直接抹除所有正常车辆前景；

3. 多帧时序校验，短暂光影抖动直接过滤，仅持续静止物体告警。

# 6 非功能性需求

## 6\.1 性能指标需求

1. 端到端总延迟：≤2秒（平板采集→前端画面告警）；

2. 单帧全套AI推理耗时：≤100ms；

3. 前端画面渲染延迟：≤500ms；

4. WebSocket消息推送延迟：≤200ms；

5. 支持并行设备：至少5台平板同时推流稳定运行；

6. 数据库单条件查询响应：≤2秒。

## 6\.2 可靠性需求

1. 系统连续稳定运行时长≥8小时无崩溃；

2. SRT 主推流断开自动重连，重连超时后切换 RTMP 备用链路，单轮重连耗时≤10 秒；

3. AI推理异常不中断整条视频链路，仅跳过当前帧；

4. WebSocket断开前端自动周期性重连；

5. 数据库写入失败不阻塞实时分析流程，仅留存日志。

## 6\.3 易用性需求

1. 前端页面零培训即可操作，按钮、面板文字清晰；

2. 设备启动自动推流、自动注册，无需复杂配置；

3. 告警信息文字直白，区分不同故障类型；

4. 训练、启动脚本一键运行，无需复杂环境配置。

## 6\.4 兼容性需求

1. 边端：安卓9及以上平板、手机通用；

2. 云端：Windows/Linux双系统，CUDA 11\.8\+；

3. 前端：Chrome、Edge主流浏览器兼容；

4. 数据集：适配沙盘小车、真实道路两类车牌画面；

5. 网络：校园局域网、手机热点、FRP内网穿透全部兼容。

## 6\.5 可扩展性需求

1. 新增摄像头仅分配device\_id，无需修改核心代码；

2. 可新增火灾、行人等AI模型，模块化接入调度；

3. 支持拓展云端多GPU分布式推理；

4. 数据库可无缝从SQLite迁移至MySQL；

5. 前端可新增统计图表、导出功能，不改动后端接口。

## 7\.1 参与者（Actor）定义

结合用例图两类角色泛化关系、各角色专属业务用例，精准划分权限与职责：

1. **前端浏览用户（基础父参与者）**基础权限角色，仅通过 PC 前端可视化页面查看系统实时数据、设备状态，无任何系统配置、设备运维、数据修改权限；系统管理员泛化继承该角色全部查看能力。对应图内用例：查看实时视频、查看车牌识别结果、查看热力图、查看违停告警、查看道路异常标注、查看设备在线状态；关联包含关系：查看设备在线状态 \<\<include\>\> 监控边缘设备；关联扩展关系：查询历史告警、查询历史事件可触发扩展用例「导出数据」。

2. **系统管理员（子参与者，泛化自前端浏览用户）**拥有系统最高全量操作权限，继承前端浏览用户所有查看功能，额外负责设备生命周期、白名单业务数据管理、模型参数配置管理；对应专属用例：注册上报设备、管理白名单、配置模型参数。

## 7\.2 用例图补充说明

1. **包含关系 \<\<include\>\> 说明**基础用例「查看设备在线状态」强制复用公共用例「监控边缘设备」；无论普通用户还是管理员查看设备状态，系统都会执行设备在线心跳查询逻辑，属于必须执行的公共流程。

2. **扩展关系 \<\<extend\>\> 说明**扩展用例「导出数据」双向关联基础用例「查询历史告警」、「查询历史事件」；

    - 基础流程：仅查询展示历史记录，属于默认必执行逻辑；

    - 扩展分支：用户有报表导出需求时才触发执行，不属于查询流程的强制步骤，仅作为可选拓展功能。

3. **参与者泛化关系说明**系统管理员与前端浏览用户为 Actor 泛化关系，实线空心三角箭头指向父参与者「前端浏览用户」；代表系统管理员**完整继承普通浏览用户所有查看权限**，同时额外拥有设备管理、白名单、模型配置等高阶管理用例权限。

# 8 数据需求

## 8\.1 数据库存储需求

采用MySQL数据库存储，核心数据表：

1. 设备信息表 edge\_device；

2. 白名单车辆表 vehicle\_whitelist；

3. 识别事件表 recognition\_event；

4. 告警记录表 alarm\_record；

5. 系统配置表 system\_config。

## 8\.2 数据字典规范

1. device\_id：设备唯一字符串标识，全局不重复；

2. plate\_number：车牌标准字符串，大写字母\+数字；

3. track\_id：车辆跟踪整型编号，单路视频内唯一；

4. bbox：图像坐标数组 \[x1,y1,x2,y2\]；

5. timestamp：统一`yyyy-MM-dd HH:mm:ss`格式时间字符串；

6. status状态枚举：online/offline/smooth/slow/congested/warning/normal。

## 8\.3 数据生命周期

1. 设备配置：永久保存，手动删除；

2. 白名单：永久存储；

3. 实时识别、告警数据：自动留存30天，过期定时清理；

4. 系统日志：保存15天；

5. 临时视频段：分析完成后立即删除，不占用磁盘。

# 9 测试需求

## 9\.1 边端采集传输测试需求

1. 采集功能测试：不同分辨率、帧率画面无模糊卡顿；

2. SRT 主推流、RTMP 备用推流稳定性测试：云端持续稳定接收视频流；

3. 设备注册/心跳测试：上线自动登记、离线状态同步；

4. 网络降级测试：弱网下调码率、分辨率正常推流；

5. 断流重连测试：断开网络重连后恢复传输；

6. 备用上传测试：关闭RTMP后短视频可正常上传云端。

## 9\.2 云端AI算法测试需求

1. 车牌识别测试：各类角度、亮度车牌识别准确率达标；

2. 车辆检测测试：沙盘小型车辆无漏检、误检率低；

3. 跟踪测试：车辆移动、交汇track\_id不跳变；

4. 违停判定测试：不同停留时长告警触发准确；

5. 道路异物测试：纸箱、石块正常告警，车辆、光影不误报；

6. 多模型并发测试：同时开启四类场景显存不溢出、无崩溃。

## 9\.3 前端可视化测试需求

1. 视频渲染：画面流畅，无明显卡顿撕裂；

2. 叠加框测试：车牌、车辆、异物坐标与实物匹配；

3. 热力图测试：车辆增减红绿黄色同步变化；

4. 告警测试：事件产生列表实时新增；

5. 控制指令测试：切换场景、修改阈值云端生效；

6. 历史查询：按时间/设备筛选数据准确。

## 9\.4 全链路联调测试需求

完整端到端流程测试步骤：

1. 平板固定，启动 SRT 主推流；

2. 云端接收视频并完成解码；

3. 云端AI多模型推理生成结构化结果；

4. WebSocket推送至前端；

5. 前端画面、热力、告警同步展示；

6. 人为制造车辆通行、拥堵、违停、路面异物；

7. 校验系统识别、告警、展示全部符合需求。

# 11 约束条件

1. 硬件约束：云端必须配备NVIDIA GPU，无GPU无法实时AI推理；

3. 性能约束：硬件算力不足时只能降低分辨率、增大抽帧间隔；

4. 时间约束：项目开发分为链路、AI、前端、联调四阶段；

5. 数据集约束：沙盘场景车牌数量有限，真实道路场景识别准确率会提升；

6. 开源约束：YOLO、LPRNet、ByteTrack均基于开源框架二次开发，无商用版权冲突。
